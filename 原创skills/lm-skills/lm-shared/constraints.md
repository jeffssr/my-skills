# LinkMed 通用约束

lm-* skill 共享的内容规范、表达规范和流程规范。一处定义，各处引用。各 skill 不再各自重复。

---

## 1. 内容约束

### 1.1 无技术细节

不出现：状态码、字段名、表名、类名、文件路径、HTTP 方法、API 路径、具体数值。

> 适用：solution.md / spec.md / prd.html 正文。纯后端需求可豁免。

| ❌ | ✅ |
|----|----|
| "订单状态从 0 变为 1" | "订单从未支付变为已支付" |
| "调用 /api/invoice/void 接口" | "作废发票" |
| "web_linkmed_clinic 表的 pay_method 字段" | "诊所的支付方式" |

### 1.2 不自加

用户没提、上游产物没支持的内容一律不写。不自加功能、页面、通知、步骤、字段。

> 适用：全部产物。

### 1.3 Portal 端不拆分

统一写"Portal 端"，不出现"诊所端 / 中介端"拆分。

> 适用：solution.md / spec.md / prd.html。

### 1.4 用户面向文案全英文

按钮、标签、弹窗提示、邮件/短信/Push 等对外文字一律英文。

> 适用：spec.md / prd.html。

### 1.5 中文产品术语

| ❌ 英文控件名 | ✅ 中文术语 |
|-------------|-----------|
| Radio / Radio Group | 单选按钮 |
| Select / Dropdown | 下拉选择框 |
| Input / Text Field | 文本输入框 |
| Placeholder | 占位提示文字 |
| Modal / Dialog | 弹窗 |
| Drawer | 抽屉 |
| Filter | 筛选 |
| Checkbox | 复选框 |
| Toggle / Switch | 开关 |
| Tooltip | 提示气泡 |
| Tab | 标签页 |
| Button | 按钮 |
| Badge | 标记 |

> 适用：spec.md。

### 1.6 交付物末尾干净

末尾不出现整改记录、审查记录、复查记录、已确认项、遗留问题等过程性章节。整改直接改正文，记录只留审查文件。

> 适用：solution.md / spec.md / prd.html。

### 1.7 待确认项不消失不假定

每条待确认项在后续阶段必须有明确去向（写入正文 / 用户确认 / 补搜解决）。不凭空假定答案，不可悄然消失。

> 适用：context.md / solution.md / spec.md。

### 1.8 核心业务流程一致

solution.md 第 3 节 = spec「一、综述 > 2 核心业务流程」。内容、粒度、措辞、顺序完全一致。spec 是复制粘贴关系，不可重写、增减、变序、改措辞。

> 适用：solution.md + spec.md。

### 1.9 正文不写 UI 设计细节

spec / PRD 正文不出现颜色、对齐、间距、圆角、阴影、字体大小等视觉属性。UI 设计稿负责视觉。原型不受此限。

> 适用：spec.md / prd.html 正文。

---

## 2. 表达约束

### 2.1 Mermaid 规范

- 默认用文字（编号列表或 → 箭头）
- 仅跨端交互或多分支（>3 个决策分支）场景用 Mermaid
- Mermaid 节点 = 业务动作 / 状态，不画 UI 交互（"点击按钮""打开弹窗"）
- PRD 的 Mermaid 以 spec 为准：spec 有 → PRD 继承；spec 文字 → PRD 保持文字；spec 无 → 不创建

> 适用：solution.md / spec.md / prd.html。

### 2.2 大白话 + 简洁

零技术细节。能用几个字说清不绕。能抽象出规则不枚举。多级无序列表代替长段落。

| ❌ | ✅ |
|----|----|
| "当用户点击该按钮后，系统将会弹出一个确认弹窗，用户需要在弹窗中确认操作" | "点击按钮 → 弹窗确认" |

> 适用：spec.md / prd.html。

### 2.3 方式选择（文 / 表 / 图）

| 条件 | 表达方式 |
|------|---------|
| 跨端交互 OR >3 个决策分支 | Mermaid 流程图 |
| 多个选项 / 映射 / 对比 | 表格 |
| 其他情况 | 文字（编号列表或 → 箭头） |

> 适用：spec.md。

---

## 3. 前置约束

### 3.1 写前 grep + Read

写涉及现有页面 / 弹窗 / 交互逻辑的内容前，必须先 grep + Read 原文件。回看：字段名、字段顺序、控件类型、原文案、原逻辑。context.md 已有信息可直接使用。功能性 UI 特征必须看代码确认，不可凭记忆。

> 适用：spec.md / prd.html。

### 3.2 材料第一

用户提供的 UI 资料优先于代码。代码与文档冲突时以代码为准。不凭经验脑补。

> 适用：全部 skill。

---

## 4. 流程约束

### 4.1 备份门禁

覆写已有产物前必须先调用 `/lm-backup` 备份。详见 `./backup-gate.md`。

> 适用：全部 skill。

### 4.2 审查由用户决定

自检通过后询问用户是否派子代理审查。不默认执行审查。

> 适用：全部 skill。

### 4.3 子代理全新上下文

派子代理时发送完整 prompt，新上下文、零共享信息。不复用历史上下文。

> 适用：lm-clarify / lm-solution / lm-review。

---

## 5. 变动标注

### 5.1 .changed 标注

| 场景 | 做法 |
|------|------|
| 全新页面 / 弹窗 / 模块 | 不加 `.changed` |
| 修改已有页面 / 弹窗 / 模块 | 整块复刻 + 改动点加 `.changed` |

> 适用：prd.html 原型。

### 5.2 修改 ≠ 重写

修改已有产物时：定位到需修改的具体段落 / 行，只改该处，其余内容一字不动。不因修改一处而连带改动周边正常内容。

> 适用：spec.md / prd.html。
