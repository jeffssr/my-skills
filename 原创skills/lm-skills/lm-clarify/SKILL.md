# lm-clarify — 需求澄清 + 代码搜索

职责：澄清不明确点 + 派子代理搜代码 + 汇总 context.md。不改源码。

---

## 触发

| 触发 | 不触发 |
|------|--------|
| 新需求起点 | 纯业务概念问题（不需搜代码也不需澄清） |
| 需求有模糊词 | |
| 交互/边界/规则未说明 | |
| 用户说"搜代码""看现有功能" | |
| context.md 不存在或已过期 | |

---

## 前置

1. 读 CLAUDE.md + 项目 memory
2. 确认需求目录（定义见 lm-backup「需求目录」章节；已继承则跳过；否则确认需求名称后按定义创建）

---

## 备份门禁

详见 `../lm-shared/backup-gate.md`。若 context.md 已存在，覆写前必须先备份。

---

## 流程

```
收到需求 → 判断不明确点？
  ├─ 有 → 步骤 A（澄清）
  └─ 无 ↓
需要搜代码？
  ├─ 是 → 步骤 B（搜代码）
  └─ 否 ↓
第8节有新待确认？
  ├─ 是 → 回到步骤 A
  └─ 否 → 步骤 C（自检）→ 步骤 D（审查决策）
```

A/B 可穿插多轮，直到无需澄清且搜索充分。

---

## 步骤 A：需求澄清

1. 读用户输入，提取核心诉求
2. 逐项检查不明确点（交互/边界/规则/范围）
3. 有不明确点 → 选项化提问

### 提问格式

推荐：Option A — <理由>

| 选项 | 描述 | 影响 |
|------|------|------|
| A / B / 其他 | ... | 选错会怎样 |

回复选项字母即可。接受推荐回复 "yes"。

### 约束

- 最多 5 轮。超限 → 标注风险，继续前进
- 不给开放式问题
- 不追问用户没提过的功能
- 先查对话历史，已有答案不重问
- 材料第一：先看用户给的文件，不凭经验脑补

---

## 步骤 B：代码搜索 → context.md

详见 `references/search-guide.md`。

核心规则：
- 主代理派子代理、汇总结果、写 context.md
- 主代理 ≠ 搜索者。子代理没搜到的 = context.md 第 8 节标注"待确认"
- 5 维度必须全部覆盖再汇总：核心功能 / 后端逻辑 / 关联影响 / 通知 / 逐端逐页
- 子代理命名：`search-r<n>-<维度>`（如 search-r1-core）
- 最多 3 轮，达上限 → 标注遗漏风险
- 子代理全新上下文，零共享

汇总模板和写入规范见 `references/context-guide.md`。

---

## 步骤 C：自检

检查维度：`references/context-guide.md` 自查清单（8 项）。

自检流程（格式、证据规则、修正循环）详见 `../lm-shared/self-check.md`。

自检文件：`<需求目录>/self-check-context.md`

修正时注意：备份门禁见 `../lm-shared/backup-gate.md`。

---

## 步骤 D：审查决策

自检通过后询问用户：

> context.md 自检通过。是否派子代理审查？
> - 是（仅内容审查）→ 调用 /lm-review（输入：context.md + 审查维度=references/context-guide.md 自查清单 + 是否包含风格审查=否）
> - 是（含风格审查）→ 调用 /lm-review（输入：同上 + 是否包含风格审查=是）
> - 否 → 等下一步指示

---

## 通用约束

内容约束和流程约束详见 `../lm-shared/constraints.md`。本 skill 特别强调：

1. **不凭经验脑补** — 无搜索依据的断言 = 违规
2. **不追问用户没提过的功能** — "要不要顺便加 XX" = 违规
3. **不改源码** — 任何源码修改 = 违规
4. **待确认项不消失** — 第 8 节条目在后续阶段无去向 = 违规
5. **主代理 ≠ 搜索者** — 主代理自己搜 = 搜索结果无效
6. **不替子代理补搜** — 主代理补搜的内容 = 无效

---

## 完成

自检 + 用户确认（含审查，如选择）后，验证 `context.md` 存在且非空。

```
lm-clarify 完成。产物：<需求目录>/context.md
如需继续：/lm-solution <需求名称>
```
