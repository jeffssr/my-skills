# solution.md 编写指南

lm-solution 产出。writer 子代理必读。通用内容约束详见 `../lm-shared/constraints.md`，本文不再重复。

---

## 定位

方案思路是产品向文档，spec 的前置参考。质量标准参考 lm-prd 的 spec-template.md 和 writing-rules.md，但更概要：
- 核心业务流程只画业务节点，不画 UI 交互
- 变更清单只列改什么
- 核心逻辑方案只写思路

第 3 节核心业务流程 = spec 第 1.2 节（完全一致，spec 是复制粘贴的关系）。

---

## 模板

```markdown
# <需求名称> — 产品方案思路

## 1. 背景
（现状是什么，问题是什么）

## 2. 需求目标
（要做什么，一句话说清）

## 3. 核心业务流程
默认用文字（编号列表或 → 箭头）。仅跨端交互或多分支场景用 Mermaid。
Mermaid 只画业务动作/状态节点，不画 UI 交互（"点击按钮""打开弹窗"）。

1. ...
2. ...
3. ...

## 4. 变更清单

| # | 位置（端 > 页面 > 模块） | 类型 | 一句话描述 |
|---|-------------------------|------|-----------|
| 1 | <端> > <页面> > <模块> | 新增/修改 | <要做成什么样> |

变更清单按操作流程顺序排列（先发生的在前），不按端分组、不按技术模块排。

## 5. 核心逻辑方案
（关键业务逻辑的设计思路，非实现细节）

## 6. 边界条件

| 场景 | 处理 |
|------|------|
| <边界场景1> | <处理方式> |

## 7. 待确认问题（必写，不可省略）

| # | 问题 | 推荐 | 选项 A | 选项 B | 影响 |
|---|------|------|--------|--------|------|
| 1 | <问题描述> | A | <方案A> | <方案B> | <选择不同会怎样> |

每个待确认项必须选项化。若 context.md 第 8 节有待确认项 → 逐条搬运。
无待确认项 → 写"无"但保留章节标题。

重要：第 7 节是暂存区。主代理必须在本阶段结束前逐条提问用户，答案写入正文。不可带入下一阶段。
```

---

## 有无 context.md 的区别

| 有 context.md | 无 context.md |
|---------------|---------------|
| 每条变更可回溯到 context | 标注"⚠ 未经代码搜索验证" |
| 第 7 节含 context 第 8 节所有待确认项 | 仅覆盖用户明确描述的功能点 |
| 自查清单中 context 相关条目全检 | context 相关条目跳过 |

---

## 自查清单

自检流程（格式、证据规则、修正循环）详见 `../lm-shared/self-check.md`。

| # | 检查项 |
|---|--------|
| 1 | (有 context.md) 变更清单覆盖了 context.md 中所有"受影响"点 |
| 2 | (有 context.md) context.md 中"不受影响"的未出现在方案中 |
| 3 | 每条一句话描述具体可执行，不是笼统的"优化XX" |
| 4 | "待确认问题"章节存在，逐条列出不确定点 |
| 5 | 无技术实现细节（API/字段名/表名/代码文件名/具体数值）。纯后端需求除外 |
| 6 | Portal 端不拆分 |
| 7 | 变更清单按操作流程顺序排列 |
| 8 | 变更范围不超过用户声明的范围 |
