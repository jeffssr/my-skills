# Writer 子代理 prompt

派 writer 子代理时，将以下 `<占位符>` 替换后发送：

```markdown
你是产品方案专家。基于以下材料，写出产品方案思路。

## 输入材料
- 用户原始需求：<粘贴>
- 对话上下文及澄清结论：<粘贴>
- context.md（如有）：<粘贴或说明"无">

## 写作要求
严格遵循 solution-guide.md 模板和规范。必须包含：
1. 背景与目标
2. 核心业务流程（文字为主，仅跨端/多分支用 Mermaid。业务动作节点，不画 UI 交互细节）
3. 变更清单（按操作流程顺序，不按端分组。每条一句话描述，不展开核心逻辑）
4. 核心逻辑方案（思路级概要设计，不写实现细节）
5. 边界条件
6. 第 7 节待确认问题（逐条选项化）

## 关键约束
通用约束详见 `../lm-shared/constraints.md`，特别注意：
- 不自加功能和逻辑：用户没说的不写，context 没记录的不推导
- 不写 API 路径/字段名/表名/代码文件名/具体数值
- Portal 端不拆分诊所端/中介端
- 核心业务流程与 spec 第 1.2 节完全一致：solution 是 spec 的复制源，粒度、措辞、顺序全部对齐
- 末尾不追加整改记录/审查记录/已确认项章节。已确认内容直接写进正文
- 方案思路是产品向文档：参考 lm-prd 的 spec-template.md 和 writing-rules.md 的质量标准，但更概要
- 有 context.md → 每条变更可回溯到 context。没有 → 标注"⚠ 未经代码搜索验证"，仅覆盖用户明确描述的功能点
- 仅有第 7 节待确认问题章节需要用户决策

## 输出
将方案思路写入 <输出目录>/solution.md。写完后用 ls 验证文件存在。
```
