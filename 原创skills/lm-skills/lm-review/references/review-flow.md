# 审查流程

## 6 步流程

```
0. 备份门禁 → 1. 派审查员 → 2. 汇总质疑 → 3. 过滤 → 4. 逐条手术整改 → 5. 复查 → 6. 判定
```

### 步骤 0：备份门禁

详见 `../lm-shared/backup-gate.md`。整改操作会覆写已有产物，进入整改前必须先备份。

### 步骤 1：派审查员

- 派 2 个独立子代理（`reviewer-r{n}-1`、`reviewer-r{n}-2`）
- 子代理条件：全新上下文、零共享信息、独立运行
- 输入：待审产物 + 审查维度（调用方提供）+ 本文件 + filter-rules.md + report-format.md
- 每个审查员输出一份审查报告，写入 `<需求目录>/review-r{n}-{1|2}.md`
- **审查员必须按 report-format.md 的字段要求输出**，缺字段的质疑项 = 无效

### 步骤 2：汇总质疑

主代理读取两份审查报告，提取所有质疑项。

汇总时丢弃缺字段的无效项。

### 步骤 3：过滤

按 filter-rules.md 逐条过滤质疑：
- 通过滤网的质疑 → 进入整改清单
- 被过滤掉的质疑 → 记录驳回理由，不整改
- "建议优化"类别 → 默认驳回

过滤结果写入 `<需求目录>/review-filter-r{n}.md`。

### 步骤 4：逐条手术整改

**整改 ≠ 重写。整改 = 定位 → 编辑 → 验证，逐条执行。**

> 已在步骤 0 执行备份门禁。若迭代多轮整改，每轮整改前均须重新备份。

#### 整改原则

1. **只改质疑项指向的具体位置，其余内容一字不动**
2. 每条质疑项独立处理，处理完后才处理下一条
3. 不允许因修改一条质疑而连带改动周边正常内容
4. 整改范围 = 仅过滤后的质疑清单，不多不少

#### 逐条整改流程

```
对整改清单中的每条质疑：
  1. 定位：找到产物中质疑指向的具体段落/行
  2. 编辑：只修改该处，替换为建议修改或自拟修正
  3. 记录：写入整改记录（见下方格式）
```

#### 整改记录格式

写入 `<需求目录>/review-fix-r{n}.md`：

```markdown
# 整改记录 - 第 N 轮

| # | 质疑# | 定位 | 修改前（原文片段） | 修改后 | 是否改动其他内容 |
|---|--------|------|-------------------|--------|----------------|
| 1 | 3 | 第4节第2段第3句 | "调用searchService.query()获取" | "根据搜索条件获取" | 否 |
| 2 | 7 | 第6节变更清单第3行 | "后端：增加审核字段" | "后端：增加审核状态字段" | 否 |

## 变更影响确认

- 修改涉及的章节数：N
- 未修改的章节数：M
- 是否存在连带修改：是/否（如有，列出具体位置和原因）
```

**"是否改动其他内容"列**：如果某条整改导致修改了非质疑指向的位置，必须在此列说明。理想状态 = 全部"否"。

#### 整改执行方式

- 派全新子代理 `writer-r{n+1}` 执行整改
- 给 writer 的 prompt 必须包含：
  1. 原始产物全文
  2. 过滤后的质疑清单
  3. **明确指令**："逐条定位修改，只改指定位置，其余内容保持原文不动。不允许重写整段/整章。修改完成后输出整改记录表。"

### 步骤 5：复查

对照整改清单 + 整改记录逐项验证：

- 质疑项是否已修正
- 修改是否仅限于质疑指向的位置
- 非质疑位置是否被误改

将结果写入 `<需求目录>/self-check-review-r{n}.md`：

```markdown
# 复查记录 - 第 N 轮

| # | 质疑# | 整改结果 | 质疑已修正 | 非质疑区域未改动 | 备注 |
|---|--------|---------|-----------|----------------|------|
| 1 | 3 | "根据搜索条件获取" | ✅ | ✅ | |
| 2 | 7 | "审核状态字段" | ✅ | ✅ | |

## 结论

- 全部通过 → 进入步骤 6
- 有质疑未修正或有非质疑区域被误改 → 再派一轮整改（同步骤 4），最多迭代 2 轮
```

### 步骤 6：判定

- 全部质疑已整改且复查通过 → 审查完成
- 2 轮整改后仍有未通过项 → 将未解决质疑提交用户决策

## 迭代上限

- 每轮审查最多 2 次整改迭代
- 超过上限仍未解决 → 提交用户，不自行决定跳过
