spec 写作规范集中于此。写 spec 前必读，写完逐项验证。通用内容约束（无技术细节、不自加、Portal 端不拆分等）详见 `../lm-shared/constraints.md`，本文不再重复。

## 写前必做

> 通用规则见 `../lm-shared/constraints.md` 3.1。以下为 spec 写作特有增补。

写涉及现有页面/弹窗/交互逻辑的 spec 内容前，必须先 grep + Read 原文件。回看：
- 原页面的字段名、字段顺序、控件类型
- 原文案（按钮、提示、弹窗内容）
- 原逻辑（流程、校验、状态变化）

context.md 已有信息可直接使用。功能性 UI 特征必须看代码确认，不可凭记忆。视觉设计细节属于 UI 设计稿范畴，spec 不写。如用户提供了 UI 资料，优先参考。用户资料与代码冲突时以用户资料为准。

## 写作原则

### 1. 大白话 + 简洁

零技术细节。能用几个字说清不绕。能抽象出规则不枚举。多级无序列表代替长段落。

- ❌ "当用户点击该按钮后，系统将会弹出一个确认弹窗，用户需要在弹窗中确认操作" → ✅ "点击按钮 → 弹窗确认"

### 2. 金字塔原理

结论先行 → 以上统下 → 分类归组 → 逻辑递进。

## 字段拆写规范

### 1. 字段拆分，不堆一行

多字段不可同行。每字段独立一行，用"上方/下方"区分层级。

- ❌ `右侧：技师姓名 + 可选时间列表 + 回复时间戳`
- ✅
  ```
  右侧：
    上方：团队负责人提交回复的时间戳（格式 dd/mm/yyyy hh:mm:ss）
    下方：技师姓名 + 可选时间（格式 HH:MM AM/PM），每条一行
  ```

### 2. 每个字段写清含义

不写"时间戳"，写"谁在什么节点产生的什么时间戳"。

### 3. 左右/上下位置对齐 proto

spec 中的"左侧/右侧""上方/下方"必须与 proto HTML 实际布局一致。写完逐项对照验证。

### 4. 文案字符串逐字对齐 proto

spec 用户面向文案必须与 proto 实际字符串逐字一致。proto 有的前缀保留，proto 没有的不加。

### 5. 列表项写明单条格式

列表型展示数据必须给出模板字符串。

- ❌ "原提交技师/时间列表" → ✅ "每条一行，格式：`{技师姓名} — {时间}`（HH:MM AM/PM）"

### 6. 变量名体现数据来源

变量用 `{角色 + 动作 + 对象}` 三要素命名。

- ❌ `{PreferredTime}` → ✅ `{患者填写的偏好时间}`

## 通知写法

通知正文内容用 markdown **代码块**包裹（spec），PRD 中对应用 `<pre>` 标签，前加 `<p><strong>内容</strong>：</p>`。按钮 `[Button Name]` 后说明点击行为。

**邮件**：发送场景 + 主题 + 收件人（角色，不写邮箱）+ 内容。
- 多个邮件变体时，每个变体各用一个 `**邮件**：<变体名>` 标签独立描述
- 多主题（首次/再次等）时，每条独立一行：`主题（xxx）：...`
- 禁用 `####` 子节表示通知变体

**短信**：只写正文。`[LinkMed]` 前缀和 `LinkMed Team` 页脚系统自动追加。

**App Push**：Title + Body（`{ServiceDate} | {ServiceAddress}`）。点击进消息详情页。

**App 消息详情页**：Title + 内容正文 + 按钮 + 点击行为。

### 通知/页面中的姓名显示

| 通知/页面针对的阅读对象 | 格式/变量写法 | 示例 |
|------|------|------|
| 非病人，但涉及病人姓名 | {PatientFirstInitial}. {PatientLast} | B. Ayala |
| 患者 | {PatientFirstName} | Brianna |
| 医生 | Dr. {DoctorLastName} | Dr. Michael |
| LinkMed 团队 | LinkMed Team / Scheduling Team | — |
| 其他 | {FirstName} | Li |

### 通知中的其他变量写法

`{PascalCase}` 语义命名，不写技术字段名。金额前加 `$`，枚举值注释可能值。

| 类型 | 示例 |
|------|------|
| 服务 | `{ServiceType}`、`{ServiceDate}`、`{ServiceAddress}` |
| 订单 | `{OrderId}`、`{InvoiceNumber}`、`{TotalAmount}` |
| 枚举 | `{BillableOrNot}` → Billable / Non-Billable |

## 修改已有功能

核心逻辑直接写修改后完整行为，不写 Before/After。原型标注写"需要（在原有基础上修改：…）"，除非用户明确要求重画。

## 多状态变体

| 差异程度 | 做法 |
|---------|------|
| 差异大（布局不同、字段差异 > 3 个、流程不同） | 按变体逐个阐述 |
| 差异小（仅局部文案变化、个别字段显隐、标记状态变化） | 整体说明，指出变化点 |

原型中用 state-label-inner 区分变体，标题与核心逻辑中的变体描述对应。

## 禁止项

| # | 禁止 | 替代做法 |
|---|------|---------|
| 1 | "场景举例"子章节 | 核心逻辑中直接覆盖所有场景 |
| 2 | "波及"子章节 | 波及内容放入对应需求点的核心逻辑 |
| 3 | 简单需求写"核心业务流程" | 非跨端且 ≤3 分支 = 简单 |
| 4 | 编造边界条件 | 先确认数据源，不同源 = 无交叉场景 |
| 5 | Mermaid 画 UI 操作 | 节点 = 业务动作/状态 |
| 6 | 操作记录用特殊颜色 | 操作记录用默认文字色，时间戳统一灰色。状态标记颜色不受限 |
| 7 | 通知变体用 `####` 子节 | 每个变体各用 `**邮件**：<变体名>` 标签独立描述 |
