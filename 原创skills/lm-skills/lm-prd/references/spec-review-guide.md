# spec 审查维度

lm-prd spec.md 专有的审查维度，供 /lm-review 调用时追加。自检与审查共用同一标准。

通用审查流程和质疑过滤规则见 `/lm-review`。审查员输出必须按 `/lm-review/references/report-format.md` 格式，缺字段的质疑项 = 无效。通用约束详见 `../lm-shared/constraints.md`，写作规范详见 `writing-rules.md`。

---

## 审查输入

调用 /lm-review 审查 spec 时，需额外提供：
- `references/spec-template.md` 全文（模板结构依据）
- `references/writing-rules.md` 全文（写作规范依据）
- `solution.md` 全文（如有；不存在则跳过相关检查项）
- `context.md` 全文（如有；不存在则跳过相关检查项）

---

## 审查维度（spec 特有）

> ★ = 关键项自检覆盖项（默认级别），无标记 = 完整自检覆盖项

### A. 模板结构

| # | ★ | 维度 | 检查点 |
|---|---|------|--------|
| 1 | ★ | 综述格式 | # 标题 → > **目标**：→ ## 一、综述 → ### 1 现状 & 问题 → 表格？现状&问题节下无额外段落？ |
| 2 | ★ | 文档结构 | 端名用 ## 一、二、...？需求点用 ### 1/2/3...（每端重新编号）？综述=一、端名递增（二、三、四...）？ |
| 3 | ★ | 需求点顺序 | 序号 需求标题 → 位置 → 触发场景 → 原型 → 核心逻辑？位置和触发场景分两行？**每个需求点均有位置、触发场景、核心逻辑标题（不可缺任何一项）**？ |
| 4 | ★ | 需求List 列顺序 | 列顺序为 `产品端 \| 需求点`？跨端写 `平台端 + Portal 端` 等形式？ |
| 5 | | 原型标注 | "需要"或"无需原型"，不写文件路径引用。修改场景允许"需要（在原有基础上修改...）" |
| 6 | ★ | 通知格式 | 通知变体用 `**邮件**：<变体名>` 标签，**禁用 `####` 子节**？多主题分独立行？内容用代码块包裹？ |
| 7 | | 变体组织 | 差异大→分变体阐述；差异小→整体说明。不在原型标注处列变体标题 |
| 8 | | 编号规范 | 端名用 ## 一、二、...，需求点用 ### 1/2/3...（每端重新编号），子节 #### 1.1/1.2...？ |

### B. 内容正确性

| # | ★ | 维度 | 检查点 |
|---|---|------|--------|
| 9 | | 方案对齐 | (有 solution.md) spec 是否遗漏 solution 变更点？是否凭空新增？ |
| 10 | | 待确认追踪 | (有 solution.md) solution.md 第7节每项在 spec 中有明确处理？无假定答案？ |
| 11 | | 完整性 | (有 context.md) context.md "受影响"点是否都体现在 spec 中？ |
| 12 | ★ | 不自加 | 用户没提、context/solution 没支持的内容？范围声明外出现 = 致命 |
| 13 | ★ | 核心流程一致 | (新需求) spec「一、综述 > 2 核心业务流程」与 solution.md 第3节逐字一致？ |

### C. 写作规范

关键项：
| # | ★ | 维度 | 检查点 |
|---|---|------|--------|
| 14 | ★ | UI 设计细节 | 正文出现颜色/对齐/间距/字体大小等 = 致命 |

完整项（deep-check 时追加）：逐条过 writing-rules.md 全部规则 + 以下核查项：
| # | ★ | 维度 | 检查点 |
|---|---|------|--------|
| 15 | | 边界条件 | 不同数据源编交叉场景？废弃功能未彻底删除？ |
| 16 | | UI 资料优先 | 用户提供了 UI 资料时 spec 是否优先参考？ |
| 17 | | PRD 同步确认 | 修改场景确认 prd.html 也将同步重新生成？ |

## 关键项自检覆盖面（★ 汇总）

**8 项**（spec 侧）：A1 综述格式、A2 文档结构、A3 需求点顺序（含四要素完备）、A4 需求List列顺序、A6 通知格式、B12 不自加、B13 核心流程一致、C14 UI设计细节

## 质疑分类枚举

模板结构 / 内容顺序 / 位置触发场景格式 / 四要素完备性 / 需求List列顺序 / 原型标注 / 变体组织 / 通知格式 / 通知标签规范 / 编号规范 / 技术术语 / 文案语言 / 不自加 / 凭空参数 / UI设计细节 / 方案对齐 / 待确认追踪 / 完整性 / 通知正确性 / 边界条件 / 写作规范 / 废弃检查 / UI资料优先 / PRD同步确认

> 以上为审查覆盖面，报告"类别"字段仍须使用 report-format.md 定义的 5 个标准类别（事实错误/规则违反/遗漏/逻辑矛盾/建议优化），此处仅用于审查员自检覆盖完整性。
