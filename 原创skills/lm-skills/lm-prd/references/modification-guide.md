# 需求文档修改指南

修改已有 spec.md / prd.html 时，同样严格遵守本技能全部规范、要求和模板。不只是改内容，规范也要检查。通用约束详见 `../lm-shared/constraints.md`。

---

## 修改铁律

1. **规范优先于内容**：修改前对照 spec-template.md、writing-rules.md、prd-guide.md、prototype-guide.md 检查现有文档。不符合的先修正，再改内容
2. **spec 和 prd 必须同步**：先改 spec.md，再改 prd.html，最后统一自检。不可只改其中一个
3. **审查统一询问**：修改场景和新建场景一致——spec 和 prd 都改完统一自检通过后询问是否审查
4. **模板要求必查**：需求点顺序？位置和触发场景分两行？CSS 只用标准模板？原型结构符合 prototype-guide.md？

---

## 修改流程选择

根据改动规模选择流程：

### 微调模式（改动 ≤3 处且不涉及结构变化）

适用条件：仅改文案、改字体颜色、改部分内容显示/隐藏等，不新增/删除需求点、不改变文档结构。

1. ⚠️ **备份门禁**：详见 `../lm-shared/backup-gate.md`
2. 改 spec.md（仅改指定处，其余内容一字不动）
3. 改 prd.html（仅改指定处，原型按下方"原型修改规则"处理）
4. **改点验证**（非全量自检）：只验证修改点与 spec 对齐、CSS 规范、.changed 标注
5. 自检通过后，询问用户是否审查

改点验证格式：
| # | 修改点 | spec 原文 | prd 对齐 | CSS/标注 |
|---|--------|----------|---------|---------|
| 1 | 第4节按钮文案 | "[Submit]" → "[Confirm]" | ✅ | ✅ |

### 标准模式（改动 >3 处或涉及结构变化）

适用条件：新增/删除需求点、重组章节、核心逻辑重写等。

流程同下方"标准修改流程"。

## 修改写作要点

### 标题语言

章节/状态标题用中文，不中英混杂、不加括号注释原文。英文用户面向标记保留在核心逻辑正文。

- ✅ "阶段二：等待患者回复"
- ❌ "阶段二：Awaiting Patient（首个团队已回复，等待患者）"

### 可读性

核心逻辑用多级无序列表代替长段落。按关键维度分层（标记/可编辑性/内容展示/按钮），子场景独立阐述。

### 内容归属

无 UI 载体的纯文字说明（校验提示、规则描述）写入核心逻辑正文。原型不为此类内容创建独立 state-block。

## 原型修改规则

| 场景 | 做法 |
|------|------|
| 用户未要求重画原型 | 在原有基础上修改：改文案、改字体颜色、改部分内容显示/隐藏等。不动整体布局和结构 |
| 用户明确要求重画原型 | 按 prototype-guide.md 规范和标准 CSS 模板重新绘制 |

判断标准：
- "改一下XX文案""XX颜色改成红色""XX 改成隐藏" → 在原有基础上改
- "重画原型""重新画""原型重做" → 重画

---

## 标准修改流程

1. ⚠️ **备份门禁**：详见 `../lm-shared/backup-gate.md`
2. 读现有 spec.md + prd.html，对照各模板/规范检查格式与规范差距
3. 改 spec.md（先修正规范问题，再改内容）
4. 改 prd.html（先修正规范问题，再改内容，原型按上述规则处理）
5. 统一自检 spec + prd → 写自检文件（含证据）→ 全 ✅ 通过

自检按 SKILL.md 步骤 3 执行（分级自检或完整自检）。自检流程详见 `../lm-shared/self-check.md`。

6. 自检通过后，一次性询问用户是否审查（spec + prd）

### 修改方式：逐条手术

修改 ≠ 重写。修改时：
- 定位到需修改的具体段落/行
- 只改该处，其余内容一字不动
- 不因修改一处而连带改动周边正常内容
- 修改后在自检证据列中标注"仅修改第N节第M段，其余未动"
